# 🛡️ 슬래시 커맨드 구현 프로토콜 (COMMAND_IMPLEMENTATION_PROTOCOL.md)

이 문서는 사용자의 "커맨드를 만들어줘"라는 요청에 대해 에이전트가 실패 없이 완벽한 결과물을 내기 위해 준수해야 할 표준 절차이다.

---

### 1. 실패 원인 분석 (Post-Mortem)
- **지식 업데이트 지연**: 에이전트가 Gemini CLI의 최신 기능(커스텀 커맨드 지원)보다 범용적인 텍스트 처리 방식에 의존함.
- **실행 환경 오판**: 터미널 UI의 자동완성 메커니즘을 과소평가함.
- **검증 절차 누락**: 커맨드 생성 후 실제 CLI가 이를 로드할 수 있는 경로와 규격인지 확인하지 않음.

### 2. 재발 방지 실행 지침 (Standard Operating Procedure)

#### [Phase 1: 요구 사항 분석]
- 사용자가 원하는 커맨드의 **Trigger(/kb:name)**, **Description**, **Prompt Logic**을 명확히 정의한다.
- 기존 스킬(`skills/*.md`)과의 연동이 필요한지 확인한다.

#### [Phase 2: 규격 기반 구현]
- 반드시 `.gemini/commands/` (또는 지정된 `.agents/` 경로) 하위에 `.toml` 파일로 생성한다.
- `description`과 `prompt` 필드를 필수로 포함하며, `{{args}}`와 같은 변수 활용을 검토한다.

#### [Phase 3: 자동 검증 (Validation)]
- 생성된 파일의 경로가 CLI 로드 규칙에 맞는지 확인한다.
- `cat` 명령으로 파일 내용을 읽어 문법 오류(TOML syntax)가 없는지 검사한다.
- 사용자에게 `/commands reload`를 실행하도록 안내하여 즉시 반영을 유도한다.

### 3. 에이전트 자가 체크리스트
- [ ] 이 커맨드는 `/` 입력 시 자동완성 리스트에 나타나는가?
- [ ] Gemini CLI의 `cli_help`를 통해 최신 커맨드 명세를 확인했는가?
- [ ] 단순 텍스트 지침이 아닌, 실제 시스템 인프라(TOML)에 반영되었는가?
