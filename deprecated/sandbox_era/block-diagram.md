현재 구축된 OpenClaw Native Sandbox 아키텍처를 도표로 정리해 드립니다. 
이 구조는 성능이 필요한 LLM은 호스트에서 직접 실행하고, 위험할 수 있는 도구 실행은 Docker 내부에 격리하는 하이브리드 모델입니다.

## OpenClaw 시스템 실행 구조도

```
[ 호스트 환경: Mac Mini (macOS) ]
┌─────────────────────────────────────────────────────────────────────────┐
│  (사용자 레이어)                                                        │
│   [ Control Web UI / CLI ] ◄───┐                                        │
│             │                  │ (HTTP/WS)                              │
│             ▼                  │                                        │
│  (코어 레이어)                 │                                        │
│   [ OpenClaw Gateway ] ────────┘                                        │
│   (Node.js 프로세스 - 배경 실행)                                        │
│             │                                                           │
│             ├───────► [ Ollama / Gemma4 ] (로컬 LLM - 호스트 GPU 활용)  │
│             │                                                           │
│             └───────► [ Docker Engine ] (컨테이너 관리)                 │
│                           │                                             │
│                           ▼ (자동 생성/실행)                            │
│             ┌───────────────────────────────────────────┐               │
│             │ [ openclaw-sandbox ] (Docker 컨테이너)    │               │
│             │ ───────────────────────────────────────── │               │
│             │  (도구 실행 레이어 - 격리 구역)           │               │
│             │   - bash / python / ls / curl 등          │               │
│             │                                           │               │
│             │  (보안 제약 사항)                         │               │
│             │   - Network: NONE (네트워크 완전 차단)    │               │
│             │   - Root FS: READ-ONLY (시스템 수정 불가) │               │
│             │   - User: Non-root (권한 제한)            │               │
│             └───────────────────────────────────────────┘               │
└─────────────────────────────────────────────────────────────────────────┘
```

## 구성 요소별 작동 방식

1. OpenClaw Gateway (Host)
   - 역할: 두뇌 및 관제 센터. 사용자 채팅 수신, 모델 라우팅, 샌드박스 생명주기 관리를 담당합니다.
   - 특징: 호스트 시스템에서 직접 실행되므로 UI와 모델 서버(Ollama)에 빠르게 접근할 수 있습니다.

2. Ollama / Gemma4 (Host)
   - 역할: 사고 및 추론.
   - 특징: GPU 자원을 직접 사용해야 하므로 샌드박스 밖(호스트)에서 실행됩니다. Gateway와 로컬 네트워크(127.0.0.1:11434)로 통신합니다.

3. openclaw-sandbox (Docker)
   - 역할: 에이전트의 "손과 발"(Tool Execution).
   - 특징: 에이전트가 "파일 목록을 보여줘" 또는 "코드를 실행해"라고 요청하면 Gateway가 이 컨테이너를 띄워 내부에서 명령을 실행합니다.
   - 연결: Docker 소켓(/var/run/docker.sock)을 통해 Gateway가 명령을 주입하고 결과를 돌려받습니다.

## 작업 흐름 (Workflow Example)
1. 사용자: "현재 폴더 파일 목록을 보여줘." (Web UI)
2. Gateway: 메시지를 받아 Gemma4(Host)에게 전달.
3. Gemma4: "파일 목록 확인을 위해 ls 도구를 써야겠어."라고 응답.
4. Gateway: Docker(Sandbox) 컨테이너 내부에서 ls 명령 실행.
5. Docker: 격리된 환경 내에서 결과 반환. (호스트 파일 시스템에는 접근이 제한됨)
6. Gateway: 결과를 다시 Gemma4에게 전달하여 최종 답변 생성.
7. 사용자: 채팅창에서 결과 확인.

이 구조 덕분에 에이전트가 실수로 호스트의 중요한 파일을 삭제하거나 외부로 데이터를 유출하는 것을 Docker 레벨에서 원천 차단할 수 있습니다.
